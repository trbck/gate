# gate — the outer loop

boil answers: *"is this feature done?"*
gate answers: *"is this **project** done — and should I even be working on it?"*

The failure mode gate exists to kill: months of vibe coding produce many
projects that are all 80% done, none productive, none profitable, because
(a) "done" was never defined observably at the project level,
(b) adding features is cheaper than finishing, so scope grows sideways,
(c) attention is split across too many projects with no forcing function
to ship, charge, or kill.

gate fixes this with four mechanisms:

1. **A maturity ladder per project** (L0–L5) where each level is a quality
   gate of *observable, evidence-backed* criteria. No evidence, no checkmark.
2. **Todo rules** that make it structurally hard to work on anything that
   doesn't advance the current gate.
3. **A session protocol** every LLM session must follow (read state → work →
   write state), so progress is measured in *gate deltas*, not vibes.
4. **A portfolio layer** with a hard WIP limit and forced park/kill decisions,
   so finite attention concentrates until something actually ships.

## File layout

Per project (created by `gate init`):

```
<project>/.gate/
├── charter.md     # WHY: goal, user, business/life value, north star, kill criteria, non-goals
├── ladder.md      # WHAT: L0–L5 gate criteria with evidence slots (the quality gates)
├── todo.md        # NOW (≤5) / NEXT / ICEBOX / DONE — every NOW item maps to a gate criterion
├── log.md         # append-only session log: date, what happened, gate delta, evidence
└── scorecard.md   # GENERATED by audits — never hand-edit
```

Workspace level:

```
/home/trbck/workspace/PORTFOLIO.md   # generated by scripts/gate-portfolio.py + weekly review notes
```

Templates for all files: `templates/` next to this SKILL.md. A filled example:
`examples/`.

## The maturity ladder (canonical)

Every project climbs the same six levels. `ladder.md` customizes the criteria
per project, but the *meaning* of each level is fixed — do not water it down:

| Level | Name | Meaning — the gate question |
|---|---|---|
| **L0** | Spark | Charter written. Deliberate decision this is worth pursuing over existing projects. |
| **L1** | Skeleton | The core loop works end-to-end once, on the dev machine. Happy path demoable. |
| **L2** | Usable by me | I actually use it for its real purpose, repeatedly (≥3 uses in one week). No data-loss bugs. Setup documented. |
| **L3** | Survives a stranger | Deployed / installable without me. Handles errors, restarts cleanly, secrets out of code, core paths tested, a stranger can onboard from the README alone. |
| **L4** | Valuable to others | Business: ≥1 external user using it without prompting; feedback captured. Life-tool: 30 consecutive days of real use; measurably changed the behavior it targets. |
| **L5** | Profitable / load-bearing | Business: recurring revenue ≥ target in charter. Life-tool: removing it would hurt; maintenance loop defined. |

Rules of the ladder:

- **Levels are sequential.** You may not claim L3 work matters while L2
  criteria are open. (Exception: a security/data-loss fix is always in scope.)
- **The most important criteria are not code.** L4/L5 force distribution,
  users, and money — the things vibe coding never gets to. An LLM can't do
  these for you; the ladder's job is to keep shoving them in front of you as
  `human-action` items instead of letting another refactor happen.
- **Criteria must be observable.** "Improve reliability" is not a criterion.
  "Service survives `docker restart` with zero data loss, verified by test X"
  is.

## Evidence rules (the quality gate mechanism)

A ladder checkbox may only be ticked with an evidence line in this exact
format:

```
- [x] Core loop runs end-to-end — EVIDENCE: `pytest tests/e2e -q` → 4 passed | 2026-07-03 | auto
```

Format: `EVIDENCE: <command → result | URL | number | screenshot path> | <YYYY-MM-DD> | <auto|human>`

- `auto` = re-verifiable by running a command. `human` = requires a human
  observation (e.g. "3 mentors recruited", "Stripe shows €147").
- **Evidence TTL:** `auto` evidence older than **14 days** is STALE — the box
  stays ticked but the audit flags it and it does not count toward the gate
  until re-verified. `human` evidence for usage/revenue criteria expires after
  **30 days**.
- **No fresh verification, no claim.** Same rule as boil: run the command,
  paste the result line, then tick. Never tick from memory.

## Todo rules (anti-scope-creep)

`todo.md` has four sections. These rules are hard rules:

1. **NOW holds max 5 items.** Every NOW item names the gate criterion it
   advances, e.g. `(→ L2.3)`. An item that maps to no open criterion of the
   *current* level may not enter NOW.
2. **New ideas go to ICEBOX by default.** Feature ideas mid-session get one
   line in ICEBOX and zero minutes of work. The only exception: it unblocks a
   current-gate criterion.
3. **Bugs beat features.** Any bug that breaks a previously-passed criterion
   reopens that checkbox and jumps to the top of NOW.
4. **Items are phrased as verifiable outcomes** ("signup returns 200 and
   sends mail — verified by story-03"), never activities ("work on auth").
5. **14-day challenge:** any NOW/NEXT item untouched for 14 days must be
   done, deleted, or iceboxed at the next audit — it may not just sit.
6. **Human-action items are first-class.** "Post on r/startups", "email 5
   potential users", "set up Stripe" live in NOW like any other item. If the
   top of NOW is human-action, the LLM's job is to prepare it (draft the post,
   the email, the checklist) and then surface it loudly — not to pick a code
   item instead.

## Session protocol (what every LLM session must do)

Any agent starting work in a project with `.gate/` MUST:

**On start (before any code):**
1. Read `charter.md`, `ladder.md`, `todo.md`, and the last 3 entries of `log.md`.
2. State in one line: current level, open criteria count, and which NOW item
   this session targets.
3. If the requested work maps to no open criterion of the current level, say
   so and ask: icebox it, or amend the ladder? (Amending the ladder is
   allowed but explicit — same as boil's "goal.md is sacred".)

**On end (before the final summary):**
4. Append to `log.md`:
   ```
   ## 2026-07-03 — session
   - did: <1–3 lines>
   - gate delta: <criteria flipped, with evidence — or "none">
   - todo delta: <moved/added/closed items>
   - next: <single most important next action>
   ```
5. Tick/untick ladder boxes only per the evidence rules.
6. Update `todo.md`.

**The effectiveness metric is the gate delta.** Three consecutive sessions
with `gate delta: none` on the same project is a mandatory trigger: stop and
run `gate audit` — the project is churning, and the audit must say why
(criteria too big? wrong level? motivation gone? → split criteria, or park).

## Audit protocol (`gate audit`, automatable)

Run on demand, after every ~5 sessions, or weekly via `/schedule`. The
auditing agent:

1. Reads all `.gate/` files and recent `git log --oneline -20`.
2. **Re-verifies evidence:** re-runs every `auto` evidence command for the
   current level; marks STALE/BROKEN ones and unticks BROKEN.
3. **Drift check:** compares the last 20 commits against the charter's goal
   and non-goals. Work not serving any criterion = drift; name it.
4. **Staleness:** days since last log entry; days since last gate delta.
5. Applies the 14-day todo challenge (rule 5).
6. Writes `scorecard.md` (template in `templates/scorecard.md`) with:
   frontmatter (stage, score, health), criteria status, drift findings, and
   **exactly one recommended next action**.
7. Updates the project's row in `PORTFOLIO.md` (or runs
   `scripts/gate-portfolio.py`).

Scoring: `score = <current level> + <fraction of next level's criteria with
fresh evidence>`, e.g. `L2 + 3/7 → 2.43`. Health flags: `OK`,
`STALE-EVIDENCE`, `DRIFTING`, `STUCK` (no gate delta 3+ sessions or 14+
days), `ZOMBIE` (no log entry 30+ days while status=active).

## Portfolio protocol (the attention layer)

`PORTFOLIO.md` at the workspace root lists every project with charter
frontmatter: status, stage, score, north star, last delta. Hard rules:

1. **Max 3 projects have `status: active`.** Everything else is `parked`
   (with an explicit re-entry condition) or `killed` (with a 3-line
   post-mortem in its charter). Parked projects reject sessions: an agent
   asked to code in a parked project must point this out first.
2. **Weekly review** (schedulable): regenerate the portfolio, then for each
   active project ask: did the score move since last week? Two consecutive
   weeks without movement forces a decision — recommit (say what changes),
   park, or kill. No silent limbo.
3. **Kill criteria are set at charter time** (pre-commitment), e.g. "kill if
   no paying user by 2026-09-30". The weekly review enforces them. Killing a
   project that met its kill criteria is a success of the system, not a
   failure of the project.
4. **New project ideas** don't get a directory. They get a charter draft in
   `projectbrainstorming/` and enter the portfolio as `candidate`. A candidate
   becomes active only by displacing (parking) a current active project —
   named explicitly.

## boil integration

gate produces the goals; boil burns them down:

- Each open ladder criterion is a valid boil goal:
  `boil "<criterion text>" till "<its evidence command> is green"`.
- boil's `.boil/goal.md` success checklist for such a run should quote the
  criterion's evidence format, so a green boil run yields a paste-ready
  EVIDENCE line.
- boil's `human-action` tickets (API keys, Stripe, DNS) should also be
  mirrored into `todo.md` NOW so they survive the boil run.
- boil stories (`references/stories.md`) are the preferred `auto` evidence
  for L2/L3 user-facing criteria.

## Bootstrapping an existing project (`gate init`)

1. Read the project's README/CLAUDE.md/docs and recent git history first.
2. Interview the user with ≤10 questions to fill `charter.md` (template has
   the questions). Push hard on: who is the user, what is the north-star
   metric, what are the kill criteria, what is explicitly OUT of scope.
3. Draft `ladder.md` from the template, customized; mark criteria the project
   plausibly already meets as `[?]` — then **verify each `[?]` with real
   evidence before ticking**. Most 3-month-old vibe projects land at L1–L2
   with holes; that's expected and useful truth.
4. Seed `todo.md` NOW with the ≤5 items that close the current gate.
5. Add the governance pointer to the project's `CLAUDE.md`/`AGENTS.md`
   (snippet in `templates/claude-md-snippet.md`).
6. Log the init in `log.md` and run a first audit (scorecard).
7. **Portfolio update — automatic, part of init, never a separate manual step.**
   Running `gate init` *is* the declaration that this project is focus now, so:
   a. Set `status: active` in the charter.
   b. Run `scripts/gate-portfolio.py --check`. If the WIP limit is now
      breached, resolve it *inside the init*: ask the user which currently
      active project this one displaces (charter question 10 already asked —
      default to that answer), set that project's charter to `status: parked`
      with a one-line re-entry condition (create a stub
      `.gate/charter.md` with frontmatter only if it has none), then re-run
      the script.
   c. Init is not complete until `gate-portfolio.py --check` exits 0 (or the
      only remaining violations are explicitly acknowledged by the user).
      Report the fresh portfolio table in the init summary.

## Hard rules (summary)

1. No checkmark without fresh, formatted evidence.
2. No NOW item without a gate criterion.
3. No feature work above the current level.
4. No session without a log entry and gate-delta statement.
5. Max 3 active projects; parked projects reject sessions.
6. Two stagnant weekly reviews force recommit/park/kill.
7. Human-action items (users, money, distribution) outrank code items —
   prepare them, surface them, never silently substitute code work.
8. The charter's non-goals section is a fence: work outside it needs an
   explicit charter edit, confirmed by the user.
